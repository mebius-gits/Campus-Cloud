# 班級管理與虛擬教室更動健檢

更新日期：2026-07-20

## 結論

班級管理、每週內容、多機器建置與虛擬教室的核心流程已可進入整合測試，但環境模板、任務檔案派送與虛擬教室工作階段仍有正式上線前必須補齊的項目。

## 已確認正常

- 班級使用獨立的 `TeachingClass`、`TeachingClassStudent`、`TeachingClassMachineNode`、`TeachingClassStudentMachine`、`TeachingClassWeek` 與 `TeachingClassTaskFile` 資料模型，不再以 Group 代替班級。
- 一位學生可關聯 Client、Web、Database 等多台固定機器。
- 班級以每週固定星期、開始時間、結束時間、時區與提前開機時間產生課表。
- 每週內容只需設定「主題／任務」及任務檔案，不需要手動選擇日期。
- 建機流程會分別追蹤各環境節點，並區分待審核、建立中、部分失敗與全部就緒。
- 全部學生機器建立成功後，班級才會切換為可上課狀態。
- 班級總覽已顯示班級基本資訊、目前週次、下次上課與建機準備狀態。
- 虛擬教室已納入班級頁籤，可依學生查看其多台機器並進行監看或廣播。

## 驗證結果

- Alembic migration 可升級到最新 head。
- Backend Ruff 靜態檢查通過。
- 班級、建機、VNC 與虛擬教室相關測試共 51 項通過。
- Frontend 測試共 61 項通過。
- Frontend production build 成功。

## 尚未正式完成

### 1. 環境模板仍儲存在瀏覽器

目前課程環境模板使用 `localStorage`。更換瀏覽器、電腦或清除網站資料後，模板會消失，也無法供不同老師或裝置共同使用。正式版本應建立後端環境模板、模板節點與模板版本資料模型。

### 2. 班級沒有保存模板版本快照

班級已保存實際機器節點，但沒有保存來源模板 ID、版本與套用當下的完整快照。正式版本應在套用模板時保存不可變的版本快照，避免日後修改模板影響既有班級。

### 3. 每位學生最多三台機器僅由前端限制

模板編輯器會在三台時停用新增按鈕，但後端仍可接收超過三個機器節點的請求。後端 API 應同步驗證節點數量為一至三台。

### 4. 任務檔案尚未派送至學生機器

任務檔案已能上傳、刪除並綁定週次，但尚未具備下載 API、派送排程與實際寫入 Client、Web 或 Database 機器的流程。

### 5. 虛擬教室工作階段尚未持久化

目前 VNC 監看與廣播 session 由後端記憶體管理。後端程序重啟後 session 會消失，老師重新整理頁面時也不一定能接回原本的廣播控制狀態。正式版本可改以 Redis 保存 session 與廣播狀態。

### 6. CSV 僅能加入既有帳號

匯入 CSV 時，系統只會加入已存在的使用者；找不到的 Email 會回報失敗，不會自動建立帳號。需與校務帳號或 LDAP 流程確認預期行為。

## 安全與資料庫修正

- 任務檔案的磁碟路徑不再包含使用者提供的原始檔名，改由後端產生 UUID 儲存名稱；原始檔名只用於畫面顯示。
- 新增 `teaching_class_machine_nodes.batch_job_id` 索引 migration，使 SQLModel metadata 與 Alembic schema 一致。

## 建議優先順序

1. 將環境模板與版本快照移至後端正式模型。
2. 在後端限制每個環境模板最多三台機器。
3. 完成任務檔案下載、派送與實體檔案清理流程。
4. 使用 Redis 保存虛擬教室與廣播 session。
5. 補上從建立班級、加入學生、套用模板、建機到班級啟用的端對端測試。

## 非阻斷項目

- Frontend build 仍有部分 bundle 超過 500 kB 的提示，可後續以 lazy loading 與 code splitting 改善。
- 正式部署前必須替換開發環境預設的 `SECRET_KEY`。
- `courseOperationsStore.js` 有檔尾空白警告，不影響執行。
