# SkyLab 產品設計觀察報告

- 日期：2026-07-19
- 分析階段：Step 1 — Observation
- 文件狀態：待使用者研究資料驗證
- 分析角色：Product Designer、UX Researcher、Design System Architect

## 文件目的

本文件的目的不是評論介面美醜或直接提出 UI 方案，而是先理解 SkyLab 真正服務的工作、使用者日常任務，以及目前產品結構可能造成的認知負擔。

依照產品設計觀察流程，在 Step 1 尚未取得足夠證據前，不進入產品價值觀、設計原則、資訊架構、使用流程或 UI 設計。

## 研究範圍與限制

本次觀察的證據來源包括：

- 專案 README 與產品規格文件。
- 前端路由、側欄、首頁、申請、資源、教學與課程相關程式碼。
- 後端服務與既有產品策略文件。

目前限制：

- 工作區沒有可連線的內建瀏覽器，因此尚未實際點擊完整流程。
- 專案內沒有發現產品行為埋點、可用性測試或真實功能使用頻率資料。
- 尚未取得學生、教師與管理者訪談。

因此，本文件會區分「可由現有產品證據確認的觀察」與「需要研究驗證的頻率假設」，不把推測寫成結論。

## 一、產品真正服務的工作

SkyLab 的核心 Job To Be Done 不是「管理 Proxmox」，而是：

> 讓校園中的學生、教師與管理者，在不必理解底層虛擬化技術的前提下，取得、使用、教學與治理雲端實驗環境。

產品實際包含三條差異明顯的工作鏈：

| 角色 | 真正想完成的工作 |
| --- | --- |
| 學生 | 快速取得可用環境、進入機器完成作業、知道申請與到期狀態 |
| 教師 | 確保全班環境準備完成、在課堂中查看或協助學生、追蹤學習進度 |
| 管理者 | 處理待審核與失敗工作、維持容量和服務穩定、管理風險與權限 |

目前產品的主要結構問題是：系統比較像「所有能力的總目錄」，尚未完全成為三種角色各自清楚的工作台。

## 二、使用者最常做的事

以下是由產品能力與既有規格推導的待驗證假設，不能視為實際使用頻率結論。

### 學生

推測高頻工作：

1. 進入既有 VM、LXC 或課程房間。
2. 查看環境是否已建立、是否正在執行、何時到期。
3. 開啟主控台或終端機。
4. 在課程中閱讀任務、操作環境並提交 Flag。
5. 啟動、停止或重新啟動自己的環境。

目前學生首頁優先呈現「課程推薦」與「快速入門」，沒有先呈現「繼續上次工作」、正在使用的機器或下一個待完成任務。這可能使首頁偏向探索新內容，而不是延續日常工作。

### 教師

推測高頻工作：

1. 選擇今天要上的班級或課程。
2. 確認多少學生已就緒，以及哪些環境異常。
3. 查看學生畫面或接管協助。
4. 查看本週任務完成狀態。
5. 執行少量批次調整。

### 管理者

推測高頻工作：

1. 處理待審核、開通失敗與卡住的工作。
2. 查看節點容量與服務異常。
3. 處理即將到期或已到期的資源。
4. 回應教師或學生的資源問題。

## 三、可能屬於偶爾使用的功能

依工作性質判斷，下列功能通常不應和每日工作取得相同導航權重：

- LDAP、治理政策與系統設定。
- IP、Gateway、Firewall 與 Reverse Proxy 設定。
- 配額規則調整。
- Audit Logs、背景任務與部署日誌。
- 模板建立與版本維護。
- GPU 管理。
- 建立、還原與刪除快照。
- 規格調整。
- AI API 金鑰與流量治理。

這些功能不是不重要，而是多數屬於低頻、高風險或特定角色工作。它們應在有需要時進入，不應持續佔據所有使用者的注意力。

## 四、操作頻率與產品結構的落差

目前首頁與側欄反映出兩種不同的產品定位：

- 首頁偏向學生的課程探索與快速建立環境。
- 側欄偏向跨角色的完整系統功能目錄。

因此，高頻工作與低頻管理能力被放在相近的導航層級。使用者進入系統後，必須先理解系統有哪些模組，再判斷自己應該去哪裡完成工作。

## 五、使用者最容易迷失的位置

### 1. 角色沒有真正分流

目前側欄約有 30 個末端入口。除了三個 AI 管理入口外，多數入口沒有依角色隱藏，因此學生也可能看見資源管理、審核、網路、系統管理、監控與教學管理入口。

這會造成兩個問題：

- 使用者必須自行判斷「這是不是我能使用的功能」。
- 使用者可能點進頁面後才被權限拒絕，增加挫折與不信任。

### 2. 教學概念重疊

目前同時存在：

- 班級管理。
- 群組。
- 虛擬教室。
- 教學面板。
- 課程學習。
- 課程管理。
- 上課機器模板。

使用者必須先理解系統內部資料模型，才知道「我今天要上課」應該去哪裡。

其中「班級管理」與「上課機器模板」直接以「待開發」狀態出現在主要導航，會讓尚未可完成的工作與已可使用的工作競爭注意力。

### 3. 申請與資源的狀態關係不清楚

「我的申請」負責顯示審核、開通、失敗和撤銷；「我的資源」又會呈現仍在開通中的 placeholder。使用者可能產生以下疑問：

- 申請核准後應留在申請頁等待，還是前往資源頁？
- 「已核准」和「已開通」有何差異？
- 開通失敗應在哪裡重試？
- 為什麼同一件事會出現在兩個地方？

目前後端的 `approved` 狀態需要前端再結合 `vmid` 與 `provisioning_status` 解讀，顯示產品的使用者狀態模型尚未完全一致。

### 4. 「申請審核」名稱重複

側欄同時存在：

- 資源申請審核。
- AI API 申請審核。

兩者在不同分類下使用相同名稱。使用者必須依賴分類上下文才能分辨工作對象。

### 5. AI 服務與雲端治理混淆

「AI 服務」目前包含 AI API、金鑰與用量治理；另一套 AI 能力則用於模板推薦、PVE 查詢和教學分析。這些能力服務不同的工作，卻都以 AI 作為分類語言。

使用者可能難以判斷「AI 服務」究竟是：

- 使用 AI 模型。
- 申請 AI API。
- 由 AI 協助管理雲端。
- 由 AI 分析教學情境。

### 6. 資源詳情層級過深

每項資源固定提供六個 tab：

- 總覽。
- 監控。
- 規格。
- 快照。
- 操作紀錄。
- 進階設定。

對多數學生而言，後四項可能是低頻或受限工作，但它們持續與總覽、監控保持相同層級。

## 六、哪些資訊不需要一直顯示

可能不需要持續顯示：

- 所有角色共用的完整系統導航。
- 「待開發」功能入口。
- Audit Logs、背景任務與部署日誌。
- LDAP、治理、Gateway 等基礎設施設定。
- 資源節點、建立時間、內部 VMID 等技術資訊。
- 快照、規格與進階設定等低頻操作。
- 沒有待辦時仍固定存在的審核與異常入口。
- 使用者不具權限的管理功能。
- 未經個人化資料支持的固定課程推薦。

目前首頁文案宣稱課程推薦是「根據你的學習歷程精選推薦」，但程式碼使用固定課程列表。若實際沒有個人化機制，這項資訊可能建立錯誤期待。

## 七、哪些資訊較可能需要持續存在

對學生：

- 正在進行的課程或工作。
- 環境的可用、建立中、失敗或異常狀態。
- 到期時間。
- 主控台或終端機入口。
- 下一個可執行動作。

對教師：

- 今天的班級與課程。
- 學生環境的就緒與異常狀態。
- 尚未完成或需要協助的學生。
- 觀看、協助或批次處理入口。

對管理者：

- 尚未處理的高風險事件。
- 待審核與逾期申請。
- 建立失敗或卡住的工作。
- 節點容量與服務健康狀態。

## 八、Step 1 初步結論

SkyLab 最大的觀察不是功能不足，而是產品同時服務三種角色，卻仍使用一套功能分類導航承載所有工作。

目前產品要求使用者先理解系統結構，才能開始完成任務。若未先確認不同角色的日常工作頻率，直接重整 UI 很可能只是重新排列功能，無法真正降低認知負擔。

## 九、完成 Step 1 前需要確認的研究問題

1. 目前主要使用者比例大約是多少？學生、教師、管理者各占多少？
2. 三種角色登入後，一週內最常完成的前三件事各是什麼？
3. SkyLab 的主要場景是平日自行使用、固定課堂教學，還是競賽／研究資源管理？
4. 教師實際如何理解「群組、班級、課程、虛擬教室、教學面板」？它們是不同概念，還是歷史功能重疊？
5. 是否有真實畫面、測試帳號、點擊紀錄、客服問題或使用者訪談可供驗證？

## 十、後續階段

待上述問題取得足夠資訊後，依序進行：

1. Step 2：萃取產品價值觀。

# 第一階段 UI 產品設計方向

- 確認日期：2026-07-19
- 優先角色：學生、教師
- 優先任務：取得環境、日常使用資源
- 範圍限制：本階段不處理管理者治理中心，也不重新設計所有低頻管理頁

本節是在角色與優先任務已確認後建立的第一階段設計方向。尚未取得真實行為數據的部分，仍需在可用性測試中驗證。

## 十一、UI 性格

### 11.1 使用者第一眼應感受到什麼

使用者應該感受到：

> 系統知道我目前的角色、正在進行的工作，以及下一步可以做什麼。

這不是用歡迎文案或裝飾營造，而是透過以下決策實現：

- 登入後直接顯示與角色相關的工作，不展示完整系統能力。
- 第一個可操作項目是「繼續工作」或「取得環境」，不是推薦內容或管理分類。
- 狀態旁直接提供下一步，例如「開啟終端機」、「查看建立進度」、「處理 3 位未就緒學生」。
- 技術資訊不消失，但延後到詳情或進階區域。

### 11.2 產品應建立的信任感

SkyLab 應建立四種可被具體感知的信任：

1. **狀態可信**：建立中、可使用、停止、失敗、即將到期等狀態必須一致，不能因頁面不同使用不同說法。
2. **結果可預期**：執行建立、停止、重新啟動或歸還前，清楚說明會發生什麼及所需時間。
3. **錯誤可恢復**：失敗狀態必須提供原因、可否重試，以及需要找誰協助。
4. **權限可信**：不向使用者展示無權限或尚未開放的主要功能入口。

### 11.3 產品類型定位

SkyLab 第一階段應定位為：

- **以專業工具為核心**：狀態清楚、操作位置穩定、支援重複使用與鍵盤操作。
- **帶有協作平台能力**：教師與學生圍繞班級和環境協作，但協作不應掩蓋主要工作。
- **包含管理系統能力**：複雜設定存在於次層，不主導學生與教師的第一層體驗。

它不應被設計成消費產品式的內容探索首頁，也不應以大量推薦、促銷式 Banner 或視覺展示作為主要互動。

### 11.4 UI 性格關鍵詞與設計行為

| UI 性格 | 具體設計決策 |
| --- | --- |
| 沉著 | 使用低彩度中性色作為大面積背景；只有狀態與主要操作使用強調色 |
| 明確 | 一個畫面維持一個主要動作；按鈕以動詞描述結果 |
| 可預期 | 相同狀態、位置與操作在不同頁面保持一致 |
| 任務導向 | 導航與標題使用使用者語言，不使用內部模組名稱作為主要入口 |
| 可恢復 | 錯誤訊息附帶原因與下一步，不只顯示失敗 Toast |
| 長時間舒適 | 控制密度、對比與動效；避免大面積高飽和色、玻璃效果與持續動畫 |

## 十二、角色任務模型

### 12.1 學生任務鏈

學生進入系統後，依優先順序通常需要回答：

1. 我現在有沒有可使用的環境？
2. 如果有，我如何立即進入？
3. 如果還在建立，要等多久、是否出錯？
4. 如果沒有，我應該取得哪一種環境？
5. 這個環境何時到期？

因此學生工作台的第一層不應是課程推薦，而應是「目前工作」。

### 12.2 教師任務鏈

教師進入系統後，依優先順序通常需要回答：

1. 我今天要處理哪個班級？
2. 全班環境是否已準備完成？
3. 哪些學生沒有環境、未開機或發生錯誤？
4. 我能否批次處理，或進入單一學生環境協助？
5. 完成處理後，班級是否已經可以上課？

因此教師工作台的第一層應是「班級就緒狀態」，不是模板管理或完整資源清單。

## 十三、導航原則與建議架構

導航應依角色產生，而不是先列出所有產品模組再做權限拒絕。

### 13.1 學生導航

第一層建議只保留：

1. **工作台**：目前課程、進行中的環境、建立進度與到期提醒。
2. **我的環境**：可使用、建立中、已停止和即將到期的環境。
3. **學習**：已加入的課程與任務；不是全站內容推薦。
4. **申請紀錄**：研究申請、被拒絕與歷史紀錄。可置於「我的環境」次層。

帳號與外觀設定放在使用者選單，不佔主要導航。

學生不應看見資源管理、審核、模板管理、網路、系統管理、背景任務與 Audit Logs。

### 13.2 教師導航

第一層建議保留：

1. **工作台**：今天的班級、環境就緒狀態與需要處理的例外。
2. **班級**：學生、環境、課堂協助與進度的統一入口。
3. **教學內容**：學習路徑、房間、任務與題目。
4. **環境模板**：教師準備課程環境時才進入。
5. **我的環境**：教師自己的測試或示範環境。

「虛擬教室」、「教學面板」與「群組」不再平行出現在第一層，應成為班級內的工作模式：

- 班級／成員。
- 環境。
- 上課監看。
- 學習進度。
- 教學設定。

### 13.3 導航行為規則

- 顯示使用者能完成的工作，不顯示無權限入口。
- 「待開發」功能不進入正式導航。
- 導航標籤優先使用工作名詞，例如「我的環境」，不使用技術模組名詞。
- 同一工作只保留一個主要入口；其他頁面以情境連結導向。
- 導航最多兩層，第二層只在進入工作域後出現。
- 深層頁面使用麵包屑或返回上下文，不在側欄增加更多永久入口。

## 十四、Layout 與閱讀順序

### 14.1 全域頁面骨架

每個主要頁面採固定閱讀順序：

1. **工作上下文**：頁面名稱、目前班級或環境。
2. **現在需要處理的內容**：正在進行、失敗、即將到期或未就緒。
3. **主要操作**：放在對應項目旁，不集中在遠離內容的工具列。
4. **支援資訊**：規格、配額、歷史與技術細節。

頁首不固定放大型歡迎文案。若標題已能說明工作，就不再增加副標與裝飾區塊。

### 14.2 學生工作台

主要焦點：立即繼續目前工作。

建議閱讀順序：

1. 需要立即注意的狀態，例如即將到期、建立失敗或教師協作邀請。
2. 目前使用中的環境，每個環境提供一個情境主要操作。
3. 建立中的環境與明確進度。
4. 沒有環境時才顯示「取得環境」。
5. 已加入課程的下一個任務。

不應固定顯示：

- 靜態課程推薦輪播。
- 所有可用模板的完整型錄。
- 沒有異常時的空白警告區。
- 大量用量統計小卡。

### 14.3 教師工作台

主要焦點：確保班級可以順利上課。

建議閱讀順序：

1. 今天或近期的班級。
2. 班級環境就緒摘要，例如 `28/30 已就緒`。
3. 需要處理的例外清單，例如 2 位建立失敗、3 位未開機。
4. 「進入班級」作為主要操作。
5. 建立模板或批次設定等準備工作延後顯示。

班級摘要不應拆成大量 Dashboard 小卡。使用一個可掃描列表，每列呈現班級、時間、就緒狀態與下一步即可。

### 14.4 我的環境

資源列表應優先支援掃描與操作，而不是讓每個資源成為資訊密集 Card。

建議每列第一層只呈現：

- 環境名稱與用途。
- 可使用、停止、建立中、失敗或即將到期。
- 到期時間或建立進度。
- 主要操作：開啟終端機／主控台，或啟動。
- 次要操作選單。

CPU、RAM、Disk、節點、VMID 和 IP 放在展開資訊或詳情頁；只有 IP 是目前工作必要條件時才提升層級。

## 十五、資訊層級

### 15.1 學生取得環境

| 層級 | 資訊 |
| --- | --- |
| Level 1 | 環境用途、可否立即建立、使用期限、主要動作 |
| Level 2 | 模板說明、預設規格、容量提示、建立進度 |
| Level 3 | 節點選擇、網路、GPU、進階規格、內部識別資訊 |

### 15.2 學生日常使用資源

| 層級 | 資訊 |
| --- | --- |
| Level 1 | 狀態、到期時間、終端機／主控台、啟動 |
| Level 2 | IP、基本規格、停止／重啟、課程或申請來源 |
| Level 3 | 監控歷史、快照、操作紀錄、規格申請、進階設定 |

### 15.3 教師班級環境

| 層級 | 資訊 |
| --- | --- |
| Level 1 | 班級、上課時間、就緒人數、異常人數、進入班級 |
| Level 2 | 學生個別狀態、環境模板、批次啟動、觀看與協助 |
| Level 3 | 個別規格、監控歷史、腳本分發、資源調整與技術日誌 |

Level 3 不應使用高飽和 Badge、主要按鈕或固定展開區域搶走 Level 1 的注意力。

## 十六、學生取得環境流程

### 16.1 第一步：先選目標，不先填規格

入口問題應是「你現在需要什麼環境？」並提供三種工作語意：

1. **開始課程實驗**：從已加入的課程進入，環境由課程決定。
2. **快速練習**：選擇模板，立即建立固定三小時環境。
3. **申請研究環境**：選擇日期與需求，送交審核。

三種路徑不應使用同一張複雜表單，也不應要求學生先理解 `quick_template`、`research`、VM 或 LXC 等系統名詞。

### 16.2 快速練習流程

1. 選擇用途或模板。
2. 顯示預設環境、固定三小時期限與目前容量。
3. 按下「建立環境」。
4. 在原位置顯示建立進度，不把使用者丟到不相關的申請表格。
5. 完成後主要按鈕變為「開啟終端機」或「開啟主控台」。

預設規格由模板決定。只有系統確實允許調整時才提供「調整規格」，並放在次層。

### 16.3 研究環境流程

研究申請採三階段：

1. **用途與期間**：說明要做什麼、開始與結束日期。
2. **環境建議**：由模板或 AI 建議規格，學生可檢視必要選項。
3. **確認送出**：摘要顯示期限、規格、審核需求與送出後會發生什麼。

AI 不應以浮動聊天視窗成為另一條流程。它應嵌入「用途與期間」階段，將自然語言需求轉為可檢查的建議，並清楚顯示哪些內容由 AI 建議、哪些由使用者修改。

### 16.4 完成後的自然下一步

- 快速練習：完成建立後直接進入環境。
- 課程實驗：返回任務內容，環境在同一工作上下文內開啟。
- 研究申請：前往單一申請狀態頁，而不是只回到混合歷史列表。

## 十七、教師取得與準備環境流程

教師不應從通用模板管理開始，而應從班級情境開始：

1. 選擇班級。
2. 系統顯示學生人數、目前環境與未就緒原因。
3. 若尚未設定環境，選擇既有課程環境模板。
4. 預覽每位學生機器數與全班資源需求。
5. 執行建立或排程。
6. 以例外優先方式顯示進度，只突出失敗與未完成項目。
7. 全部就緒後，主要動作變為「進入上課監看」。

模板建立、PVE 母機、網路拓撲與版本設定屬於準備工作的次層。教師只在需要新增或修改模板時進入。

## 十八、日常使用資源流程

### 18.1 學生

進入「我的環境」後：

1. 系統依需注意程度排序：失敗／即將到期 → 建立中 → 執行中 → 已停止。
2. 執行中的環境直接提供「開啟終端機／主控台」。
3. 停止中的環境直接提供「啟動」。
4. 建立中的環境顯示目前階段與預估等待，不提供無效操作。
5. 失敗環境顯示原因、重試條件或聯絡教師。
6. 使用完成後，停止、歸還或回到課程任務是自然下一步。

### 18.2 教師

教師從班級的「環境」分頁操作：

1. 預設以學生為列顯示環境狀態。
2. 系統先列出未就緒、停止或失敗學生。
3. 批次操作只在選取學生後出現，避免固定佔據畫面。
4. 單一學生列提供觀看、協助與詳細資訊。
5. 正常學生保持低視覺權重，避免綠色 Badge 填滿整頁。

## 十九、元件設計原則

### 19.1 Environment Row

解決問題：讓使用者快速掃描環境狀態並執行下一步。

必要內容：名稱、狀態、時間、主要動作、次要選單。不是每項資料都需要 Icon，也不需要每列獨立陰影 Card。

### 19.2 Status Indicator

解決問題：讓使用者理解目前發生什麼以及能做什麼。

狀態文字使用完整語意：

- 建立中 · 正在複製模板。
- 可使用 · 2 小時 18 分後到期。
- 已停止 · 可重新啟動。
- 建立失敗 · 可重試。

顏色是輔助，不作為唯一辨識方式。

### 19.3 Inline Progress

解決問題：讓非同步建立流程保持可見，不要求使用者在多頁之間尋找結果。

只顯示可理解的業務階段，例如「排隊中、建立環境、啟動中、可使用」，不暴露內部 Job ID。

### 19.4 Exception List

解決問題：讓教師先處理阻礙上課的少數例外。

正常狀態以摘要呈現，異常項目才展開。避免為每個學生建立一張同等權重的狀態卡。

### 19.5 Empty State

解決問題：告訴使用者為什麼沒有內容，以及下一步。

例如「你目前沒有環境」後直接提供「取得環境」，而不是只有裝飾插圖或要求使用者自行前往申請頁。

## 二十、Design System 規則

### 20.1 Typography

採用少量、穩定的文字層級：

| 用途 | 建議規格 |
| --- | --- |
| Page title | 24px／32px，600；每頁僅一個 |
| Section title | 18px／26px，600；只有內容確實需要分段時使用 |
| Item title | 15px／22px，600 |
| Body | 14px／22px，400 |
| Secondary | 13px／20px，400 |
| Metadata | 12px／18px，500；不承載主要資訊 |

規則：

- 主要資訊不使用全大寫英文或過度 letter spacing。
- 中文內文避免小於 13px。
- 數值狀態使用 tabular numerals，方便掃描時間與資源數字。
- 字重主要使用 400、500、600，避免每層都加粗。

### 20.2 Color

顏色依功能使用，不依頁面自由選擇：

- Neutral：背景、文字、邊界與結構。
- Primary：目前頁面的單一主要操作與焦點。
- Success：工作已完成或環境可使用。
- Warning：即將到期、容量緊張或需要注意。
- Danger：失敗、不可逆或資料風險。
- Info：建立中、審核中等非錯誤進行狀態。

限制：

- 狀態必須同時有文字或圖形，不只依賴顏色。
- 大面積背景不使用高飽和狀態色。
- 正常狀態不需要大量綠色 Badge；只在掃描確實有幫助時使用。
- 主要文字與背景至少符合 WCAG AA 對比要求。

### 20.3 Spacing

以 4px 為基準，採固定階梯：`4、8、12、16、24、32、48`。

- 元件內部：8–12px。
- 同一資訊群組：12–16px。
- 不同工作區段：24–32px。
- 頁面主要上下留白：32px；不為追求空曠任意放大。
- 密集資料表可使用 40–48px 列高；操作列不因裝飾拉高。

### 20.4 Button

| 層級 | 用途 |
| --- | --- |
| Primary | 畫面目前唯一主要任務，例如建立環境、開啟終端機 |
| Secondary | 取消、返回、檢視或不改變主要狀態的操作 |
| Tertiary | 低頻輔助操作，以文字按鈕呈現 |
| Danger | 刪除、歸還等不可逆或高風險操作 |

規則：

- 同一工作區域原則上只有一個 Primary。
- 按鈕使用結果導向文字，例如「建立環境」，不使用模糊的「確定」。
- Icon 僅在能加速辨識時加入；一般文字按鈕不必固定搭配 Icon。
- 破壞性操作不與主要前進操作並列，放入次要選單或確認流程。

### 20.5 Form

- 欄位依使用者決策順序排列，不依資料庫欄位分類。
- 標籤常駐，不以 placeholder 代替 label。
- 使用合理預設值，並說明預設值的影響。
- 驗證靠近欄位顯示；提交後也保留全頁錯誤摘要。
- 必要欄位保持可見；進階規格與技術設定收合。
- 多階段表單顯示目前階段與完成條件，不製造過多細碎步驟。
- AI 建議以可編輯建議呈現，清楚標註來源，不自動覆寫使用者輸入。

### 20.6 Navigation

- 依角色與工作顯示，不依後端模組顯示。
- 最多兩層；同一工作不產生多個平行入口。
- 當前位置需穩定可辨識。
- 使用者設定放在帳號選單。
- 低頻管理功能放入次層「管理」或只對具權限者顯示。
- 未完成、無權限或實驗功能不進入正式導航。

### 20.7 Feedback

非同步操作使用四層回饋：

1. 操作開始：按鈕立即進入處理狀態，防止重複提交。
2. 處理中：在原工作位置持續顯示進度。
3. 完成：更新內容並提供自然下一步。
4. 失敗：保留上下文，顯示原因、可否重試與支援方式。

Toast 只作為補充，不承載必須記住的結果。建立失敗、審核拒絕與即將到期等重要狀態必須留在頁面上。

### 20.8 Motion

- 只用於解釋狀態改變、展開收合或確認操作結果。
- 一般介面轉場控制在 120–200ms。
- 建立中不使用持續吸引注意的裝飾動畫。
- 支援 `prefers-reduced-motion`。

## 二十一、第一階段應移除或降級的元素

若以刪除至少 20% 非必要元素為目標，第一階段可優先處理：

1. 從學生側欄移除所有管理、審核、網路與日誌入口。
2. 從正式導航移除「待開發」項目。
3. 移除學生首頁的固定歡迎副標與靜態課程推薦輪播。
4. 將申請與建立狀態整合進「我的環境」，申請歷史降為次層。
5. 資源列表第一層移除節點、VMID、建立時間與完整規格。
6. 將 Audit Logs、快照、規格與進階設定從學生資源詳情的同級主要 tab 降級。
7. 將教師側欄的群組、班級、虛擬教室與教學面板收斂為「班級」。
8. 移除正常狀態的重複 Badge、裝飾 Icon 與無操作價值的 Card 容器。
9. 不在沒有異常、邀請或進度時保留空白區塊。
10. 不以 Toast 取代頁面內的持續狀態。

## 二十二、最終自我檢查

### 使用者是否知道下一步？

學生先看目前環境；沒有環境時取得環境。教師先看班級就緒與例外，再進入班級處理。

### 是否存在不必要資訊？

現況存在大量跨角色導航、技術資訊與低頻功能。第一階段設計已明確列出降級與移除範圍。

### 是否為了漂亮犧牲效率？

建議不依賴輪播、大型 Banner、過度 Card、玻璃效果或動畫；以列表、狀態與近端操作為主。

### 新增功能後是否容易擴充？

新增能力應加入既有工作域的次層，不再新增側欄平行分類。例如新的課堂協助功能加入「班級」，不是新增另一個教學入口。

### 是否能長時間使用？

使用中性色、大面積低彩度、固定密度、有限動效與一致狀態語言，適合長時間操作。

### 如果刪除 20% 元素，體驗是否更好？

是。現況的主要改善不是加入更多元件，而是刪除跨角色入口、重複教學概念、低頻同級 tab 與沒有下一步的資訊。

## 二十三、第一階段成功指標

後續應以任務結果驗證設計，而非只做主觀視覺評估：

- 學生從登入到開啟既有環境的中位時間。
- 學生第一次取得快速環境的完成率與中途離開位置。
- 學生是否能正確解釋建立中、已核准、可使用與到期狀態。
- 教師判斷班級是否可上課所需時間。
- 教師找到未就緒學生並完成處理所需步驟。
- 每個角色誤入無權限頁面的比例。
- 因找不到環境、申請狀態或連線入口產生的支援請求數。

這些指標需要在實作前加入最小必要埋點，才能驗證資訊架構是否真的降低工作成本。
